# Commit 最佳實踐（精簡版）

本文件提供撰寫 Git commit 的精簡規範，讓歷史可讀、可追溯、易維護。

## 核心原則
- 原子性：一個 commit 只做一件事，避免混雜無關變更。
- 可讀性：訊息讓他人不用看程式碼也能理解變更意圖。
- 可追溯性：必要時連結到 issue/任務編號。
- 可回滾性：每個 commit 都應能獨立回復或回滾。

## 訊息格式建議
- 主旨（Subject）不超過 50 字元，使用祈使語氣（例：新增、修正、移除）。
- 主旨首字小寫或依團隊規範一致，結尾不加句號。
- 主旨與本文之間空一行。
- 本文每行建議不超過 72 字元。
- 本文描述「為何」與「如何」，避免只寫「做了什麼」。

## 推薦範例
```
新增登入頁的錯誤提示

當 API 回傳 401 時顯示錯誤訊息，避免使用者重試失敗。
關聯：#123
```

## 團隊規範（可選：Conventional Commits）
若團隊採用 Conventional Commits，可使用：
- 格式：`type(scope): subject`
- 常見 type：`feat`、`fix`、`docs`、`refactor`、`test`、`chore`

範例：
```
feat(auth): 新增登入失敗提示
```

## 常見地雷
- 使用「update」「fix bug」等過於籠統的主旨。
- 一個 commit 同時含功能、重構、格式化。
- 把未完成的實驗（WIP）直接合併進主分支。
